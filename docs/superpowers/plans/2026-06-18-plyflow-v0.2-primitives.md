# plyflow v0.2 — Control-Flow Primitives Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Add the control-flow primitives plyflow needs for dynamic multi-agent orchestration — `if`, `loop`, `foreach` (with dependency waves), `agent-sdk` agentic mode, model overrides, and nested journaling — without breaking v0.1 behavior.

**Architecture:** Extract a recursive `runSteps` executor from `engine.ts` so composite step types (`loop`, `foreach`) run child sub-pipelines through one shared path. Conditionals and nested journaling become engine-level concerns; `agent-sdk` extends the Claude provider.

**Tech Stack:** Node 20 ESM, TypeScript strict, Zod v4, Vitest, `@anthropic-ai/claude-agent-sdk`.

## Global Constraints

- Node >= 20; ESM; intra-project imports use `.js` extension.
- TypeScript strict; source under `src/`; tests co-located `*.test.ts`.
- TDD: failing test first, then minimal code, then commit.
- Expressions `${{ }}` resolved against `{ inputs, steps, env }`, extended with `item` (foreach) and `iteration` (loop).
- All v0.1 tests must continue to pass after every task.
- Conventional Commits.

## Task 1: Extract recursive `runSteps` executor (engine refactor, no behavior change)

**Files:**
- Create: `src/core/exec.ts`
- Modify: `src/core/engine.ts`
- Test: `src/core/exec.test.ts`

**Interfaces:**
- Produces:
```ts
export interface ExecScope {
  inputs: Record<string, unknown>;
  env: Record<string, string | undefined>;
  baseDir: string;
  provider: AIProvider;
  registry: StepRegistry;
  outputs: Record<string, unknown>;        // accumulates this scope's step outputs
  bindings: Record<string, unknown>;        // extra expr context: { item?, iteration? }
  journal: Journal;
  journalPath: string;                      // composite prefix, e.g. "phase:Build/foreach:Apollo"
  dirty: Set<string>;
  emit(e: EngineEvent): void;
  prompt(stepId: string, req: PromptRequest): Promise<unknown>;
  runChildren(steps: StepDef[], extraBindings: Record<string, unknown>, subPath: string): Promise<Record<string, unknown>>;
}
export async function runSteps(steps: StepDef[], scope: ExecScope): Promise<Record<string, unknown>>;
```
- `runSteps` orders `steps` via `planPhase`-style waves (reuse `scheduler.planPhase` by wrapping steps as a synthetic phase), runs each wave concurrently, and for each step does the v0.1 per-step work (resolve `with`/`prompt`, hash, journal cache check with dirty-cascade, retry, emit). The expression context is `{ inputs, steps: <scope outputs as {output}>, env, ...bindings }`.

- [ ] **Step 1: Write the failing test** — `src/core/exec.test.ts`: build an `ExecScope` over a `FakeProvider` with a 2-step pipeline (`a` runs `return 1`, `b` needs `a`, runs `return ctx.steps.a.output + 1`), assert `runSteps` returns `{ a: 1, b: 2 }` and emits step-start/step-done for both.
- [ ] **Step 2: Run** `npx vitest run src/core/exec.test.ts` → FAIL (no module).
- [ ] **Step 3: Implement** `src/core/exec.ts` by moving the per-step + wave logic out of `engine.ts`'s `runWorkflow` into `runSteps(steps, scope)`. The expression context builder uses `scope.bindings` (spread last so `item`/`iteration` are visible). Then refactor `runWorkflow` in `engine.ts` to: build the root `ExecScope` (journalPath `phase:<name>` per phase), and call `runSteps(phase.steps, scope)` per phase, threading `scope.outputs` so cross-phase `${{ steps.x }}` keeps working. `runChildren` constructs a child scope sharing journal/provider/registry/emit but with its own `outputs`, merged `bindings`, and nested `journalPath`.
- [ ] **Step 4: Run** `npx vitest run` → ALL v0.1 tests + the new test PASS (engine behavior unchanged).
- [ ] **Step 5: Commit** `refactor(core): extract recursive runSteps executor`.

> NOTE: This is the riskiest refactor. Keep `runWorkflow`'s public signature, `EngineEvent` shapes, journaling hash inputs, and resume semantics identical — only the internal structure changes. The whole v0.1 suite is the regression gate.

## Task 5: `foreach` step type (dynamic fan-out + dependency waves)

**Files:** Create `src/steps/foreach.ts`. Modify `src/core/types.ts` (`foreach?`, `as?`, `key?`, `dependsOn?`, `concurrency?` on `StepDef`), `src/core/format-schema.ts`. Register in `buildDefaultRegistry`. Test: `src/steps/foreach.test.ts`.

**Interfaces:** `makeForeachStep(): StepType` — matches `def.foreach !== undefined`. Resolves the array, builds element nodes `{ key, dependsOn }`, topo-sorts into waves (reuse a generalized version of `scheduler.planPhase` — extract `planNodes(nodes: {id, needs}[])` if helpful), runs waves with a concurrency cap, each element via `ctx.runChildren(steps, { item }, 'foreach:<key>')`. Output = map keyed by element `key` → that element's sub-step output map. Unknown `dependsOn` or cycle throws.

- [ ] **Step 1: Write the failing test** — `foreach` over `[{n:'A',d:[]},{n:'B',d:['A']}]` with `key: "${{ item.n }}"`, `dependsOn: "${{ item.d }}"`, child step `echo` (`run: "return ctx.with.item ? ctx.with.item.n : ctx.bindings.item.n;"` — align to the real `item` binding) ... assert output keyed `{ A: {...}, B: {...} }` and that A's wave precedes B (assert via emitted step-start order or a shared counter). Also a test that a cycle throws `/cycle/i` and an unknown dep throws `/unknown/i`. Also a concurrency test: 3 independent elements with `concurrency: 2` never exceed 2 in flight (track via a counter step).
- [ ] **Step 2: Run** → FAIL.
- [ ] **Step 3: Implement** — add the `foreach` fields to `StepDef` + schema; implement `makeForeachStep` using the wave planner + a concurrency-limited runner (a simple promise-pool). Expose `item` in the child bindings. Register in `buildDefaultRegistry`.

- [ ] **Step 5: Commit** `feat(steps): add foreach dynamic fan-out with dependency waves`.
